System Architecture

Most analytics features follow one chain: UI component -> server action wrapper -> AI flow -> validated output schema -> chart or card rendering. The wrapper layer in src/lib/actions.ts centralizes fallback behavior and keeps UI components simple.

Core Path

  1. User creates/updates tasks in local storage-driven UI.
  2. Analytics component prepares compact task payloads.
  3. Component calls a handle* wrapper in src/lib/actions.ts.
  4. Wrapper calls a Genkit flow from src/ai/flows/*.ts.
  5. Flow runs prompt + schema validation and returns structured output.
  6. Wrapper returns flow output or fallback output.
  7. Component renders chart/cards from the returned structure.

1) Create Schedule

Primary file: src/ai/flows/create-schedule.ts

Purpose: Converts natural language planning intent into structured schedule text and a task list with actionable entries.

Input Contract

{
  "scheduleDescription": "string"
}

Output Contract

{
  "scheduleText": "string",
  "tasks": [
    {
      "name": "string",
      "description": "string?",
      "category": "string?",
      "priority": "low | medium | high?",
      "dueDate": "YYYY-MM-DD?"
    }
  ]
}

How It Works

  1. Reads user request as plain language schedule intent.
  2. Generates a segmented textual schedule with work and rest blocks.
  3. Extracts distinct actionable tasks from the schedule.
  4. For complex intents, breaks one large goal into multiple concrete tasks.
  5. For simple intents, returns one task or an empty task list if extraction is not meaningful.

Fallback and Edge Behavior

  • The flow uses a non-null assertion on output. Wrapper-level handling is important.
  • In src/lib/actions.ts, handleCreateSchedule normalizes missing tasks to an empty array.

Where It Is Used

Wrapper: src/lib/actions.ts -> handleCreateSchedule

2) Analyze Time Usage

Primary file: src/ai/flows/analyze-time-usage.ts

Purpose: Estimates weekly hour distribution (Mon-Sun) across Study, Work, Personal, Chill, and Sleep.

Input Contract

{
  "tasks": [{ "name": "string", "description": "string?", "dueDate": "string?", "priority": "low|medium|high?", "status": "todo|inprogress|done|blocked", "category": "string?" }],
  "currentDate": "YYYY-MM-DD"
}

Output Contract

{
  "weeklyUsage": [
    { "day": "Mon|Tue|Wed|Thu|Fri|Sat|Sun", "Study": 0, "Work": 0, "Personal": 0, "Chill": 0, "Sleep": 0 }
  ],
  "analysisSummary": "string?"
}

How It Works

  1. Identifies target week from currentDate (Monday start).
  2. Maps tasks into day-level estimates using category and due-date context.
  3. Returns all seven days in strict order.
  4. Adds assumptions for Chill and Sleep when not explicitly inferable from tasks.
  5. Completes missing days with default values if model returns partial data.

Fallback and Edge Behavior

  • If model output is empty, flow returns a full-week fallback with default Chill/Sleep.
  • Wrapper handleAnalyzeTimeUsage adds another safe fallback if flow call fails.

Where It Is Used

UI consumer: src/components/analytics/time-usage-chart.tsx

Wrapper: src/lib/actions.ts -> handleAnalyzeTimeUsage

3) Calculate Efficiency Score

Primary file: src/ai/flows/calculate-efficiency-score.ts

Purpose: Produces a 0-100 efficiency score with a qualitative message and optional feedback.

Input Contract

{
  "tasks": [{ "name": "string", "dueDate": "string?", "priority": "low|medium|high?", "status": "todo|inprogress|done|blocked", "category": "string?" }],
  "currentDate": "YYYY-MM-DD"
}

Output Contract

{
  "score": 0,
  "message": "string",
  "positiveFeedback": "string?",
  "improvementSuggestion": "string?"
}

How It Works

  1. Evaluates completion ratio across task set.
  2. Penalizes overdue tasks and blocked tasks.
  3. Treats overdue high-priority work as a stronger penalty.
  4. Generates contextual score narrative and optional guidance text.

Fallback and Edge Behavior

  • If no tasks exist, returns score 0 with a no-task message.
  • If flow output is unavailable, returns score 50 fallback in flow.
  • Wrapper may return score 0 if runtime error occurs.

Where It Is Used

UI consumer: src/components/analytics/efficiency-score.tsx

Wrapper: src/lib/actions.ts -> handleCalculateEfficiencyScore

4) Predict Burnout

Primary file: src/ai/flows/predict-burnout.ts

Purpose: Predicts burnout risk as low/medium/high with a 0-100 progress value and explanatory factors.

Input Contract

{
  "tasks": [{ "name": "string", "dueDate": "string?", "priority": "low|medium|high?", "status": "todo|inprogress|done|blocked", "category": "string?" }],
  "currentDate": "YYYY-MM-DD"
}

Output Contract

{
  "riskLevel": "low | medium | high",
  "progressValue": 0,
  "message": "string",
  "contributingFactors": ["string?"]
}

How It Works

  1. Evaluates active task load (todo + inprogress).
  2. Checks overdue quantity and deadline density.
  3. Weights high-priority pressure and blocked-state friction.
  4. Combines signals into risk level and mapped progress range.

Fallback and Edge Behavior

  • No tasks: low risk and low progress value by default.
  • Missing model output: medium risk default with generic message.
  • Wrapper-level catch also returns medium risk safety output.

Where It Is Used

UI consumer: src/components/analytics/burnout-predictor.tsx

Wrapper: src/lib/actions.ts -> handlePredictBurnout

5) Analyze Life Balance

Primary file: src/ai/flows/analyze-life-balance.ts

Purpose: Categorizes tasks across life areas and identifies balance gaps.

Input Contract

{
  "tasks": [{ "name": "string", "description": "string?", "dueDate": "string?", "priority": "low|medium|high?", "status": "todo|inprogress|done|blocked", "category": "string?" }],
  "timeframe": "string?"
}

Output Contract

{
  "categories": [{ "name": "string", "count": 0, "percentage": 0, "description": "string", "color": "string?", "tasks": ["string"] }],
  "totalTasks": 0,
  "balanceInsight": "string",
  "recommendation": "string"
}

How It Works

  1. Classifies tasks into life categories (work, health, relationships, growth, fun, finance, home).
  2. Counts tasks per category and computes percentages.
  3. Adds category descriptions and sample task names.
  4. Generates an insight about over/under concentration and a recommendation.

Fallback and Edge Behavior

  • Flow can return an empty category set when analysis fails.
  • Wrapper handleAnalyzeLifeBalance triggers buildLifeBalanceFallback when needed.
  • Fallback uses inferLifeCategory keyword matching from src/lib/actions.ts.

Where It Is Used

UI consumer: src/components/analytics/life-balance-chart.tsx

Wrapper and fallback logic: src/lib/actions.ts -> handleAnalyzeLifeBalance, inferLifeCategory, buildLifeBalanceFallback

6) Dynamic Task Reallocation

Primary file: src/ai/flows/dynamic-task-reallocation.ts

Purpose: Reschedules tasks to new dates after disruptions such as illness, conflict, or missed time blocks.

Input Contract

{
  "reason": "string",
  "currentTasks": [{ "name": "string", "dueDate": "YYYY-MM-DD", "duration": 0 }]
}

Output Contract

{
  "rescheduledTasks": [{ "name": "string", "newDueDate": "YYYY-MM-DD" }],
  "summary": "string"
}

How It Works

  1. Reads disruption reason to capture scheduling constraints.
  2. Reviews due dates and durations for current task set.
  3. Moves tasks to future slots while balancing load.
  4. Returns concise summary of what changed and why.

Fallback and Edge Behavior

  • Flow uses non-null assertion. If model output fails, wrapper protection should be added where integrated.
  • Dates are expected in YYYY-MM-DD and should be normalized before calling flow.

Where It Is Used

Primary logic file: src/ai/flows/dynamic-task-reallocation.ts

7) Intelligent Task Breakdown

Primary file: src/ai/flows/intelligent-task-breakdown.ts

Purpose: Splits one complex task into smaller sub-tasks with estimated durations.

Input Contract

{
  "task": "string"
}

Output Contract

{
  "subTasks": [{ "name": "string", "estimatedTime": "string" }]
}

How It Works

  1. Reads large task statement, including implicit deadline pressure.
  2. Decomposes into logical sequential units.
  3. Assigns approximate effort per sub-task.
  4. Returns simple checklist-friendly output.

Fallback and Edge Behavior

  • No explicit fallback in flow file; integration should guard against null output.
  • Quality depends on input specificity. Vague input yields generic sub-tasks.

Where It Is Used

Primary logic file: src/ai/flows/intelligent-task-breakdown.ts

8) Speech/Meeting-Aware Scheduling

Primary file: src/ai/flows/speech-meeting-aware.ts

Purpose: Adjusts user schedule around an event and returns reminders plus a speaker checklist.

Input Contract

{
  "calendarEvent": "string",
  "currentTasks": "string"
}

Output Contract

{
  "adjustedTasks": "string",
  "reminders": "string",
  "speakerChecklist": "string"
}

How It Works

  1. Parses event context and current task sequence from string inputs.
  2. Compresses and reallocates tasks around event timing.
  3. Calls built-in generateChecklist tool for event prep checklist.
  4. Returns adjusted schedule text plus reminder list and checklist text.

Fallback and Edge Behavior

  • Tool-assisted flow is powerful but string-in/string-out format requires careful prompt discipline.
  • No explicit fallback object in flow file; callers should add wrapper-level safety for production robustness.

Where It Is Used

Primary logic file: src/ai/flows/speech-meeting-aware.ts

9) Progress Chart Algorithm (Client-Side, Non-AI)

Primary file: src/components/analytics/progress-chart.tsx

Purpose: Computes task status distribution and renders a donut chart from local task storage.

Input Contract

Input source: getTasksFromLocalStorage()
Task statuses counted: todo | inprogress | done | blocked

Output Contract

Chart data array:
[
  { name: "To Do", value: number, fill: color },
  { name: "In Progress", value: number, fill: color },
  { name: "Done", value: number, fill: color },
  { name: "Blocked", value: number, fill: color }
]
(only non-zero buckets are rendered)

How It Works

  1. Loads tasks from local storage in browser context.
  2. Initializes counts for each status bucket to zero.
  3. Single pass over tasks increments matching status count.
  4. Builds chart data objects from counts and removes empty buckets.
  5. Renders donut chart and percentage labels from this dataset.

Fallback and Edge Behavior

  • If no tasks exist, UI displays an onboarding hint instead of chart.
  • If chart data is empty but tasks exist, UI displays a no-status message.

Where It Is Used

Primary logic and render: src/components/analytics/progress-chart.tsx
